# Filesystem Watching

How Sprig stays responsive to filesystem changes without burning battery. The watcher feeds `RepoState`'s dirty set, drives `core.fsmonitor` so the user's `git status` is fast too (ADR 0024), and triggers shell-extension badge updates.

ADR cross-references: 0004 (FSEvents-based watcher with per-repo incremental index), 0021 (perf budget), 0022 (best-effort + polling fallback for non-local volumes), 0024 (Sprig is the single fsmonitor source of truth).

## Layered model

```
Filesystem events
       │
       ▼
┌─────────────────┐   per-platform native impl (or polling)
│  FileWatcher    │
│  (PlatformKit   │   AsyncStream<WatchEvent>
│   protocol)     │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│ EventCoalescer  │   priority-weighted dedupe over a time window
│ (PlatformKit,   │
│  pure value)    │
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  RepoState      │   per-repo dirty set + badge trie
│  (planned M2)   │
└──────┬──────────┘
       │
       ├────► shell extension (push badge updates)
       │
       └────► core.fsmonitor hook (answer git's "what changed since X?")
```

## `FileWatcher` protocol (`PlatformKit`)

```swift
public protocol FileWatcher: Sendable {
    func start(paths: [URL]) -> AsyncStream<WatchEvent>
    func stop() async
}

public struct WatchEvent: Sendable, Hashable {
    public let path: URL
    public let kind: WatchEventKind
    public let timestamp: Date
}

public enum WatchEventKind: Sendable, Hashable, CaseIterable {
    case created, modified, removed, renamed, overflow, unknown
}
```

The protocol is small on purpose. Consumers don't care which OS they're on — they `for await event in watcher.start(paths: roots)` and react. The `overflow` kind signals "the underlying watcher dropped events; treat the roots as dirty and rescan from scratch."

## Implementations

### `WatcherKit.FSEventsWatcher` (macOS, live)

CoreServices FSEvents stream. Configured with:

- `kFSEventStreamCreateFlagFileEvents` — file-level granularity (not just directories).
- `kFSEventStreamCreateFlagNoDefer` — kernel doesn't batch; we coalesce ourselves with `EventCoalescer`.
- `kFSEventStreamCreateFlagWatchRoot` — gets notified if the watched root itself disappears.
- `kFSEventStreamCreateFlagIgnoreSelf` — drop events generated by Sprig's own writes (we know about those already).
- Latency 0.25s — Apple-recommended balance between responsiveness and kernel-side coalescing. `EventCoalescer` adds another window on top for deduplication of editor-save bursts.

State (the FSEvents stream pointer + the AsyncStream continuation) is protected by `OSAllocatedUnfairLock<State>` (macOS 13+; we require 14+). The pointer itself is wrapped in a `@unchecked Sendable` `StreamHandle` because `FSEventStreamRef` is `OpaquePointer`, which Swift 6 explicitly marks non-Sendable. The wrapper acknowledges that we own the synchronization.

Flag → `WatchEventKind` mapping:

| FSEvents flag | `WatchEventKind` |
|---|---|
| `kFSEventStreamEventFlagMustScanSubDirs` | `.overflow` (rescan signal) |
| `ItemRemoved` | `.removed` |
| `ItemRenamed` | `.renamed` |
| `ItemCreated` | `.created` |
| `ItemModified` / `InodeMetaMod` / `ChangeOwner` / `XattrMod` | `.modified` |
| anything else | `.unknown` |

### `WatcherKit.PollingFileWatcher` (portable, live)

Pure-Swift periodic-rescan watcher. The default on Linux and Windows today; on macOS it's the fallback for network volumes / iCloud Drive (per ADR 0022).

- Spawns a polling Task that takes an initial snapshot, sleeps `pollInterval`, snapshots again, diffs metadata (size + mtime), yields events.
- Diff is metadata-only — no content hashing — so per-tick cost is one `stat` per file in the watched tree.
- Lifecycle: actor-isolated `LifecycleState` tracks the polling Task. `stop()` cancels and `await`s the Task's cleanup so callers can rely on stop meaning "the stream is finished." Portable across all three OSes (`OSAllocatedUnfairLock` is macOS-13+ only; the actor is universal).
- Two consequences worth flagging: (a) doesn't fire `.renamed` (would need to correlate disappear+appear pairs by hash; future work); (b) doesn't fire `.overflow` (it can't drop events — the snapshot is authoritative).

### `WatcherKit.MockFileWatcher` (portable, live, test fixture)

Buffers emits issued before `start()` so tests can queue events up front and then drain via the stream. Used in unit tests anywhere the production code wants a `FileWatcher`.

### Native `INotifyWatcher` (Linux) — stub

Will use Linux `inotify(7)` with `IN_MOVED_TO` / `IN_MOVED_FROM` to render `.renamed` natively, plus a coalescer. A future `fanotify(7)` variant could be even cheaper but requires elevated privileges. Owning milestone: post-1.0 (Linux GUI shell follow-on). Until then `PollingFileWatcher` covers Linux.

### Native `ReadDirectoryChangesWatcher` (Windows) — stub

Will use `ReadDirectoryChangesW` per directory plus IO-completion-port coalescing. Buffer-overflow recovery falls back to a full rescan (yields `.overflow`). Owning milestone: M2-Win — when the Windows shell extension lands, we want kernel-level perf parity with FSEvents.

## `EventCoalescer` (`PlatformKit`)

Pure value-type (no timers, no threads). The watcher feeds events in via `ingest(_:)`; `RepoState` calls `drain(upTo: cutoff)` at each tick.

- One stored event per path. New events for the same path replace the old when their `kind.priority` is higher; ties keep the newer timestamp.
- Priority order (descending): `overflow > removed > renamed > created > modified > unknown`. The rule: **destructive kinds win**. A late `modified` for a path we already saw `removed` doesn't overwrite the removal — `git status` would otherwise miss the deletion until the next change.
- `overflow` is special: when present, `drain(upTo:)` returns *all* stored events regardless of cutoff. Caller treats this as "rescan from scratch."

The unit tests in `EventCoalescerTests` exercise every priority interaction and the overflow-drain semantics.

Tick-driven design (vs. timer-driven inside the coalescer) keeps testing deterministic — no `Task.sleep` in the test suite, just call `ingest` then `drain` with a chosen cutoff `Date`.

## `core.fsmonitor` integration (planned M2/M3)

Per ADR 0024, Sprig is the single source of truth for filesystem state on repos it watches. Integration plan:

1. `RepoState` exposes a token-keyed query: "what paths changed since token X?" — probably implemented as a monotonically-increasing sequence number over the dirty set.
2. We register a small fsmonitor hook script with the user's git via `git config core.fsmonitor <path>`. The hook is a thin wrapper that talks to SprigAgent over IPC and answers git's query.
3. `git status` invocations from any source (terminal, our own `Runner`, a different GUI) become O(changed paths) instead of O(tree).

This is a meaningful perf win on monorepo-scale repos. Without it, we hit the budget walls in `performance.md` only because we coalesce — the user's terminal `git status` is still a full tree walk, which is awful UX.

The hook protocol itself is part of the M1 follow-up Round 9 questions (master plan §4); details get pinned down in the M2 PR that ships it.

## Performance considerations (gates from ADR 0021)

- Steady-state CPU: <2%. Achieved by FSEvents/inotify being event-driven (no polling cost) plus coalescing (tick at 100–250ms rather than per-event).
- Memory: <150 MB for the agent watching a 100k-file repo. Dirty set is path strings — ~50 bytes each on average — so 100k entries ≈ 5 MB worst case.
- Badge update latency: <100 ms after `git` writes complete. End-to-end path: git writes → FSEvents fires → coalescer holds for 100ms tick → `RepoState` updates → push badge change to FinderSync over XPC. The 100ms coalescer window is where most of the budget goes; the rest is microseconds.

Benchmarks in `tests/benchmarks/` will gate these in CI as the M1 perf-validation work lands.

## What can go wrong

- **Network volumes / iCloud Drive / NAS.** FSEvents misses events on these. Auto-fallback to polling per ADR 0022; banner in the repo view explaining reduced real-time behavior; user can disable polling per-volume (so Sprig stops scanning a slow share entirely).
- **Symlink loops.** Polling watcher walks the tree; needs cycle detection. Open issue for the polling impl; current code uses `FileManager.default.contentsOfDirectory(at:includingPropertiesForKeys:options:)` which doesn't follow symlinks by default.
- **`.gitignore` semantics.** The watcher reports *all* events, ignored or not; filtering by `.gitignore` happens in `RepoState` (which knows the repo's ignore rules) before badges are drawn or `git status` is invoked. That keeps the watcher dumb and the ignore logic centralized.
- **Buffer overflow.** FSEvents and ReadDirectoryChangesW both signal it; we map to `.overflow` and force a full rescan. The polling watcher can't overflow (it always sees the current state).
